# OAuth 账号强绑定运行身份包方案

## 需求重述

我把你的诉求整理成下面这版，更接近产品和工程都能落地的表达：

> 目标是在 Quotio 中，把每个 OAuth 账号提升为一个“独立运行主体”。  
> 每个账号必须绑定且只绑定一套专属运行身份包，身份包至少包含：
> 
> - 出口代理
> - User-Agent 指纹
> - TLS 指纹 / 传输画像
> 
> 任何请求在出站时，都必须使用该账号绑定的身份包，且不同账号之间不得复用、串用或自动回落到同一套身份包。  
> 系统还必须提供可验证机制，能证明某个账号的真实请求确实走了它绑定的身份包。

## 我对你需求的确认

基于现有代码，我先按下面这组假设推进。如果你不同意其中任一条，后续实现策略会变：

- 绑定粒度是“OAuth 账号 / AuthFile”，不是“provider 全局”。
- 一个身份包默认只能绑定一个账号；不允许多账号共享同包。
- 未绑定身份包的账号默认不允许进入真实请求链路。
- 批量导入代理时，系统可以一键为每条代理生成一套待用指纹包。
- 你要的“TLS 指纹”是严格的运行身份概念，不是仅仅改一个 HTTP header。

如果以上理解无偏差，我建议按本文方案推进。

## 对现状的关键判断

### 已确认的事实

- OAuth 账号已经以 `AuthFile` 形式存在，并具备 `auth_index`。
- 常规请求链路目前只能使用一个全局 `proxy-url`。
- 当前 UI 没有“身份包管理”与“账号绑定”页面。
- 当前日志无法证明账号和身份包的真实绑定关系。

### 工程上的真实难点

这个需求不是单纯 UI 需求，而是“运行时出站身份编排”需求。实现成败取决于下面这件事：

- 真正发往 AI provider 的那一跳，到底能不能按账号切换代理 / UA / TLS。

因此本项目需要同时覆盖三层：

1. 数据模型
2. UI 交互
3. 出站执行引擎

## 建议的产品设计

## 1. 信息架构

我建议把你说的“主界面左侧单独加一个添加和批量导入代理、请求指纹信息”的想法，收敛为下面这个结构：

- 左侧主导航新增一级页面：`身份包`
- 现有 `Providers` 页面保留为账号管理页
- `身份包` 页面负责：
  - 添加单个代理
  - 批量导入代理
  - 一键生成 UA / TLS 指纹包
  - 查看身份包库存、状态、占用关系
- `Providers` 页面负责：
  - 查看 OAuth 账号
  - 给账号绑定或更换身份包
  - 明确显示“已绑定 / 未绑定 / 绑定冲突 / 指纹失效”

这样比把所有内容塞进 `Providers` 更稳，原因是：

- 代理池和指纹池本质上是独立资源域
- 账号页应该聚焦“账号状态与绑定结果”
- 后续如果要加“回收、封禁、冷却、健康检查”，身份包页更好扩展

## 2. 身份包数据模型

建议引入两个核心实体：

### RuntimeIdentityPackage

字段建议：

- `id`
- `name`
- `status`
- `sourceType`：手工创建 / 批量导入生成
- `proxy`
  - `scheme`
  - `host`
  - `port`
  - `username`
  - `passwordRef`
- `uaProfile`
  - `id`
  - `uaString`
  - `secChUa*`
  - `acceptLanguage`
- `tlsProfile`
  - `id`
  - `mode`
  - `clientHelloTemplate`
  - `alpn`
  - `sniStrategy`
- `verification`
  - `lastVerifiedAt`
  - `lastExitIP`
  - `lastUAEcho`
  - `lastTLSDigest`
- `binding`
  - `boundAuthFileId`
  - `boundAuthIndex`
  - `boundProvider`

### AccountIdentityBinding

字段建议：

- `authFileId`
- `authIndex`
- `provider`
- `accountKey`
- `packageId`
- `bindingMode`：strict
- `createdAt`
- `updatedAt`

## 3. 严格绑定规则

必须明确成系统规则，而不是 UI 提示：

- 一个账号只能绑定一个身份包。
- 一个身份包同一时间只能被一个账号占用。
- 账号未绑定时，不允许参与真实请求。
- 绑定失效时，该账号自动进入 `blocked` 或 `unavailable` 状态。
- 批量路由、round-robin、fill-first 都不能跨身份包选路。

## 4. UI 交互草案

### 身份包页

页面建议拆成三块：

- 顶部工具栏
  - `添加代理`
  - `批量导入`
  - `批量生成指纹`
  - `批量验证`
- 左侧列表
  - 身份包卡片
  - 显示代理出口、UA 标签、TLS 标签、绑定状态、最近验证时间
- 右侧详情
  - 代理详情
  - UA 详情
  - TLS 详情
  - 当前绑定账号
  - 最近验证结果

### Providers 页

每个账号行新增：

- 当前绑定身份包 badge
- `绑定身份包`
- `更换身份包`
- `解除绑定`
- `验证绑定`

Provider 分组展开后，账号详情区建议加：

- 账号标识
- auth index
- 当前身份包
- 真实出口验证结果
- 最近一次请求证据

## 技术实现路线

## 路线判断

这里有两个实现方向。

### 路线 A：扩展 CLIProxyAPIPlus 原生支持

目标：

- 让底层代理原生理解 `auth_index -> identity package`
- 每次出站前按账号选择 proxy / UA / TLS profile

优点：

- 架构最干净
- 账号选择和出站执行在同一进程里
- 证明链更完整

缺点：

- 依赖 CLIProxyAPIPlus 支持改造
- 本仓库只是 GUI 包装层，可能需要联动上游项目

### 路线 B：引入本地 Egress Identity Gateway

目标：

- Quotio 本地再起一层“身份出口网关”
- `CLIProxyAPI` 的全局 `proxy-url` 指向这个本地网关
- 网关根据账号特征，把流量转发到该账号绑定的代理 / UA / TLS profile

优点：

- 不必立即大改 GUI 外部结构
- 可以把“身份包控制”从现有代理中解耦出来

缺点：

- 需要可靠识别当前请求属于哪个账号
- TLS 指纹控制可能需要更底层的实现，不一定能靠现有 Swift 网络栈完成

## 我的建议

优先做路线 A 的能力调研；如果 CLIProxyAPIPlus 无法短期支持，再评估路线 B。

原因：

- 你要的是“强绑定、不可混用”
- 这类约束越靠近真实出站执行层越可靠

## 验证与测试方案

## 目标

验证某个 OAuth 账号发请求时，真实使用了它绑定的身份包，而不是只在 UI 上显示“已绑定”。

## 测试分层

### 1. 配置正确性测试

检查点：

- 身份包导入后是否正确持久化
- 批量生成后的 UA / TLS profile 是否完整
- 一个账号绑定后是否阻止其他账号复用同包
- 未绑定账号是否被正确阻断

### 2. 路由决策测试

检查点：

- 发起某账号请求时，系统解析出的 `runtimePackageId` 是否正确
- 切换账号后，解析结果是否同步变化
- 并发请求下是否发生串包

### 3. 出口真实性测试

建议做三类验证：

#### 3.1 代理出口验证

方法：

- 为每个身份包分配可区分的代理出口
- 请求一个回显服务，记录出口 IP

判定：

- 账号 A 的请求只能出现出口 IP-A
- 账号 B 的请求只能出现出口 IP-B

#### 3.2 UA 回显验证

方法：

- 请求回显 `User-Agent` 和相关 header 的测试端点

判定：

- 返回值必须与绑定的 `uaProfile` 完全一致

#### 3.3 TLS 指纹验证

方法：

- 请求能返回 JA3 / TLS hello 摘要的观测端点
- 或使用自建检测服务记录 TLS 握手摘要

判定：

- 返回的 TLS 指纹摘要必须与 `tlsProfile` 对应

## 建议新增的验证模式

为了让验证可重复，我建议新增一个“身份验证”动作：

- 在账号详情点击 `验证绑定`
- 系统用该账号发起一组最小探测请求：
  - IP echo
  - UA echo
  - TLS echo
- 生成一份验证结果：
  - 时间
  - auth index
  - runtime package id
  - exit IP
  - echoed UA
  - TLS digest
  - 是否通过

## 自动化与手工验证建议

当前仓库没有成体系自动化测试，因此建议：

### 自动化

- 先补模型层与绑定规则测试
- 如果执行层可注入 mock，再补路由决策测试

### 手工验证

- 本地启动 Quotio
- 导入两条不同代理
- 生成两套身份包
- 分别绑定两个 OAuth 账号
- 用同一 provider 的两个账号各发 10 次探测请求
- 检查日志是否始终落在各自包上
- 并发压测验证是否串包

## 我建议的交付顺序

为了控制风险，我建议按下面顺序做：

1. 先完成架构能力评估，确认运行时实现路线
2. 再补身份包数据模型和绑定模型
3. 然后做 UI 与基础存储
4. 最后接真实出站执行与验证链

如果顺序反过来，先做 UI，最后很可能出现：

- 看起来支持绑定
- 实际运行时仍是全局代理

这会直接违背你的核心诉求。
