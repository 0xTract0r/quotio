# OAuth 账号运行身份绑定通用 PRD

> 最后更新：2026-03-18
> 文档类型：PRD / reference
> 适用范围：具备 OAuth 账号体系、本地代理或请求转发层、可扩展出站执行链路的类似项目

## 1. 文档目的

本文从 `oauth-account` 相关设计文档中提炼出一份项目无关的通用 PRD，用于指导类似项目实现“OAuth 账号与运行身份绑定”能力。

本文不是某个仓库的实现说明，而是需求模板。它定义目标、边界、契约、验收方式，以及哪些地方必须按项目能力裁剪。

## 2. 核心问题

在支持多个 OAuth 账号的 AI 代理、网关或桌面管理工具中，常见问题是：

- 系统能管理多个账号，但真实请求仍共用全局出口配置
- 系统能显示账号列表，但无法证明某次请求实际使用了哪个账号的运行身份
- 系统能做账号级管理操作，但普通业务流量没有账号级运行时约束

这会导致：

- 不同账号共享代理、UA 或 TLS 行为，产生身份串用风险
- 无法对账号级请求建立审计和证明链
- 无法在合规、风控或稳定性场景下把“账号”视为独立运行主体

## 3. 目标与适用边界

本 PRD 的目标是让系统支持“账号级运行身份控制”，并默认以 `strict-exclusive` 作为推荐策略。

### 3.1 默认目标

1. 系统能为 OAuth 账号分配运行身份配置。
2. 系统能在真实请求链路中按账号选择对应运行身份。
3. 系统能阻止不满足绑定策略的请求继续出站。
4. 系统能为请求产出可查询、可分级的证明信息。

### 3.2 默认策略

本文默认策略是 `strict-exclusive`：

- 一个账号同一时间只能绑定一个身份包
- 一个身份包同一时间只能绑定一个账号
- 未绑定账号默认禁止真实出站
- 不允许静默回落到全局默认出口

### 3.3 可裁剪边界

如果目标项目采用别的策略，例如共享池、区域池、临时借用包，本 PRD 仍可复用，但必须先显式替换“绑定策略”章节，再定义新的验收标准。

换句话说：

- 本文的领域模型是通用的
- 本文的默认策略不是所有项目的唯一合法策略
- 若不改策略定义，不应宣称自己实现的是本文默认模式

## 4. 非目标

除非目标项目另行扩展，本 PRD 默认不包含：

- 在第一阶段完成真实 TLS 指纹引擎
- 替换现有 OAuth 授权流程
- 通过猜测请求内容来推断账号归属
- 仅靠 UI 展示就宣称“真实强绑定已生效”
- 在没有运行时证据的前提下宣称“TLS 指纹已落地”

## 5. 核心术语

### OAuthAccount

通过 OAuth、设备码、浏览器会话或等价方式接入系统的单个账号实体。系统内必须具备稳定主键，并能关联到运行时账号索引或等价上下文。

### RuntimeIdentityPackage

账号真实出站时使用的一组运行身份配置。至少包含：

- 出口代理或等价出站路由配置
- User-Agent 或等价请求头画像
- TLS profile 或等价传输画像

可选扩展项包括：

- Accept-Language / Client Hints
- SNI / ALPN 策略
- DNS / 出口区域策略
- 健康状态与验证快照

### AccountIdentityBinding

账号与身份包之间的绑定关系。它是一条可持久化、可审计、可校验冲突的策略记录，不等于“运行时已经成功生效”。

### Runtime Account Resolution

指系统在真实请求链路中识别“当前请求实际对应哪个账号”的能力。没有这一步，就不能宣称完成账号级运行时绑定。

### Request Proof

指系统为单次真实请求记录的证明信息，用于表达：

- 计划使用了哪个账号与身份包
- 实际观察到了哪些出站事实
- 当前证据可信度到什么等级

## 6. 用户与场景

### 目标用户

- 需要管理多个 AI 账号的个人开发者
- 需要控制账号独立出口身份的团队管理员
- 需要审计账号运行行为的运维或安全角色

### 关键场景

1. 用户导入或创建多个 OAuth 账号，希望账号之间的运行身份不串用。
2. 用户批量导入代理资源，希望系统快速生成待绑定身份包。
3. 用户查看账号列表时，需要一眼识别哪些账号未绑定、冲突、失效或未验证。
4. 系统处理真实请求时，必须根据既定策略决定是否允许出站。
5. 用户或运维排障时，需要通过请求级证据判断绑定是否真实生效。

## 7. 功能需求

### 7.1 领域模型

系统至少需要以下实体：

- `OAuthAccount`
- `RuntimeIdentityPackage`
- `AccountIdentityBinding`
- `IdentityVerificationSnapshot`
- `RequestProof`

`RuntimeIdentityPackage` 至少应包含：

- 标识字段：`id`、`name`、`status`
- 路由字段：`scheme`、`host`、`port`、`credentialRef`
- 请求画像字段：`uaString`、必要请求头画像
- 传输画像字段：`tlsProfileId` 或等价描述
- 验证字段：`lastVerifiedAt`、`exitIP`、`uaEcho`、`tlsDigest`
- 占用字段：当前是否被占用、被谁占用、占用策略

`AccountIdentityBinding` 至少应包含：

- 账号主键
- 账号运行时索引或等价引用
- 身份包主键
- 绑定策略
- 生命周期状态
- 创建时间与更新时间

### 7.2 绑定策略

绑定策略必须被建模，而不是只写死在页面逻辑里。

最低应支持：

- `strict-exclusive`
- `pooled-shared` 或明确标注“不支持”

如果项目只实现 `strict-exclusive`，也应在模型和文档中把它作为显式策略，而不是隐含假设。

默认 `strict-exclusive` 的规则如下：

1. 一个账号同一时间只能绑定一个身份包。
2. 一个身份包同一时间只能绑定一个账号。
3. 未绑定账号不能进入真实出站链路。
4. 已绑定身份包在解绑前不能被删除或分配给其他账号。
5. 绑定冲突必须在持久化层与运行时都被拒绝，而不只是 UI 提示。

### 7.3 账号解析契约

这是需求能否成立的前置条件。

系统必须定义“账号从哪里解析出来”，并明确优先级与失败策略。允许的来源可以包括：

- 已选定的 auth token / auth slot
- 已建立的 provider session
- 由可信内部组件传入的运行时上下文
- 出站执行层原生解析出的账号索引

必须禁止的来源包括：

- 根据请求 body、prompt、model 名称做启发式猜测
- 根据目标域名、路径、参数做账号推断
- 使用未受信任的外部 header 直接决定账号

账号解析失败或解析结果不唯一时，默认策略应为 `fail closed`，而不是自动回落到共享默认身份。

### 7.4 身份包管理

系统应支持：

- 创建单个身份包
- 编辑身份包基础信息
- 批量导入代理并生成身份包
- 批量生成或补全 UA / TLS 配置
- 查看身份包可用、已绑定、失效、阻断等状态
- 删除未绑定且满足安全条件的身份包

敏感信息要求：

- 代理密码、密钥或凭据不得明文保存在普通模型字段中
- 应保存凭据引用或安全存储句柄

### 7.5 账号侧管理

系统应在账号视图中直接暴露：

- 当前绑定状态
- 绑定身份包
- 更换身份包
- 解绑身份包
- 最近一次验证结果
- 最近一次请求证据摘要

账号侧至少应显示：

- 账号标识
- 运行时索引或等价上下文
- 当前绑定包
- 绑定策略
- 绑定状态
- 验证状态

### 7.6 运行时接线

真实请求链路必须具备以下能力：

1. 在常规业务请求链路中解析当前请求实际使用的账号。
2. 根据账号解析结果查到唯一适用的绑定关系。
3. 根据绑定关系选择身份包，并应用其中的代理、UA、TLS 或等价配置。
4. 当绑定缺失、账号不可解析、身份包不可用或策略不满足时，拒绝请求并返回明确错误。
5. 不允许静默退化为共享全局代理、共享默认 UA 或共享默认 TLS 行为，除非当前项目明确采用非严格策略且文档已改写。

如果目标项目做不到第 1 到第 3 条，则只能宣称“完成了模型和管理基础设施”，不能宣称“运行时强绑定已生效”。

### 7.7 证明与验证

系统必须把“配置存在”和“真实生效”区分开。

`RequestProof` 至少应包含：

- 账号主键
- 账号运行时索引或等价上下文
- 身份包主键
- 代理出口标识
- 出口 IP
- UA profile 标识
- TLS profile 标识
- 请求时间
- proof level
- proof status
- trace / correlation id

proof level 建议至少分为：

- `configured`
  - 仅证明系统内存在绑定关系和待用配置
- `route-selected`
  - 证明运行时已选择该身份包进行出站准备
- `egress-observed`
  - 证明系统已观察到实际出站使用的关键事实，例如出口 IP 或回显 UA
- `third-party-verified`
  - 证明外部验证端点或可信第三方观测到了该次请求的身份特征

系统应显式区分：

- 本地配置已存在
- 运行时已选择
- 运行时已观察
- 外部已验证

### 7.8 失败语义

系统必须定义最小失败语义，至少包括：

- `account_unresolved`
- `binding_missing`
- `binding_conflict`
- `identity_unavailable`
- `proof_unavailable`

每类失败至少应明确：

- 是否阻断真实出站
- 返回给调用方的错误类别
- 是否允许自动重试
- 是否需要人工恢复
- 是否写入审计或告警

### 7.9 UI / 交互要求

建议至少具备两个视图域：

- 账号管理视图：聚焦账号、策略和绑定状态
- 身份包管理视图：聚焦资源池、导入、编辑、验证

UI 必须明确区分以下状态：

- `draft`
- `available`
- `bound`
- `verificationFailed`
- `blocked`
- `unbound`

UI 禁止把“已配置”误导呈现为“已生效”。

## 8. 非功能需求

### 正确性

- 绑定关系必须具备一致性约束
- 删除、换绑、导入等操作必须可回滚或可恢复

### 安全性

- 敏感凭据使用安全存储
- 账号不可解析、未绑定、冲突或身份不可用时默认禁止出站，除非策略另有定义

### 可审计性

- 关键操作应保留时间、对象、策略与结果
- 请求级证据应可查询、可关联、可区分可信等级

### 可扩展性

- 身份包模型应允许未来增加更多传输层画像字段
- 运行时接口应可与外部代理、网关或上游服务集成

## 9. 外部依赖边界

对类似项目而言，最终是否能实现“真实运行时绑定”，通常取决于出站执行层是否支持：

- 在普通请求链路中识别当前账号
- 按账号或绑定策略选择代理
- 按账号或绑定策略注入 UA 画像
- 按账号或绑定策略切换 TLS 画像
- 返回请求级证明字段，或允许外部验证

如果这些能力依赖外部代理、中间层或上游服务，项目必须：

- 在 PRD 中显式标注依赖项
- 在阶段结论中区分“本仓库已完成”和“依赖外部系统”
- 禁止把依赖未落地的能力写成已交付事实

## 10. 分阶段实施建议

### Phase 0：能力调研

目标：

- 确认出站层是否能识别账号并切换身份配置
- 确认 TLS 能力是原生支持、可扩展，还是暂不可行
- 定义绑定策略、账号解析契约和失败语义

阶段完成标准：

- 已明确运行时账号解析来源与失败策略
- 已明确当前项目采用哪种绑定策略
- 已明确哪些能力依赖外部系统

### Phase 1：模型与本地管理

目标：

- 建立身份包与绑定模型
- 落地本地持久化与安全存储
- 完成账号侧与身份包侧管理 UI

阶段完成标准：

- 可以维护账号、身份包、绑定关系
- 可以清晰区分配置存在与运行时生效
- 不能对外宣称“真实运行时绑定已完成”

### Phase 2：运行时接线

目标：

- 在真实请求链路中解析账号
- 按绑定关系切换代理 / UA / TLS 或等价配置
- 补齐失败阻断与错误回传

阶段完成标准：

- 未绑定、冲突、不可解析等状态会被正确阻断
- 普通请求链路已按绑定关系选择运行身份
- 至少达到 `route-selected` 级别证明

### Phase 3：证据链与运维能力

目标：

- 记录请求级证明字段
- 增加验证动作、验证快照与排障视图
- 支持健康检查、封禁、冷却、失效恢复等运维状态

阶段完成标准：

- 至少部分请求可达到 `egress-observed`
- 运维可以区分配置、选择、观察、验证四种状态
- 文档和 UI 都不会夸大证明等级

## 11. 总体验收标准

以下条件全部满足时，才能认为默认 `strict-exclusive` 模式完成：

1. 系统可以稳定维护账号、身份包、绑定关系三类核心数据。
2. 任一账号都能在 UI 和存储层看到唯一绑定结果。
3. 账号不可解析、未绑定或绑定冲突时，真实请求会被阻断，而非静默回落。
4. 运行时出站确实按绑定关系使用代理、UA、TLS 或等价配置。
5. 系统能为请求产出可查询、可分级的证明字段。
6. 文档与 UI 都不会把“仅完成本地配置”误报为“真实运行时绑定已生效”。

如果项目采用非 `strict-exclusive` 策略，应重写第 11 节，而不是直接沿用。

## 12. 实施禁区

以下做法应被明确禁止：

- 用多个全局代理配置轮换来冒充账号级绑定
- 在桥接层通过猜测请求内容来决定账号归属
- 使用未受信任的 header 直接驱动账号选择
- 在未接通运行时之前对外宣称 TLS 指纹已落地
- 只做 UI 绑定关系展示，不做运行时阻断和证据链，却宣称需求完成

## 13. 复用方式

迁移到其他项目时，优先替换以下项目特有名词：

- `AuthFile` 替换为目标项目的账号实体名
- `auth_index` 替换为目标项目的运行时账号索引
- `CLIProxyAPI` / `ProxyBridge` 替换为目标项目的出站执行层或网关组件
- `IdentityPackage` 保持为通用领域概念即可

复用本 PRD 前，必须先回答三个问题：

1. 真实请求链路在哪一层识别账号？
2. 哪一层真正决定代理、UA、TLS 的出站行为？
3. 哪一层能够产出可信的请求级证据？

这三个问题若没有明确答案，项目只能先做到 Phase 0 或 Phase 1，不能直接宣称进入完整交付。
