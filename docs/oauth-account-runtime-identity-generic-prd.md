# OAuth 账号独立运行身份通用 PRD

> 最后更新：2026-03-18
> 文档类型：PRD / reference

## 1. 核心需求

每个 OAuth 账号都必须被视为一个独立运行主体。

系统需要为每个账号绑定一套专属运行身份，并确保该账号的所有真实请求都只能使用这套身份出站。运行身份至少包含：

- 出口代理
- User-Agent 画像
- TLS 指纹或等价传输画像

## 2. 要解决的问题

很多类似项目虽然支持多个 OAuth 账号，但真实请求仍共用全局出口配置。结果是：

- 不同账号会串用代理或指纹
- 无法证明某个账号的请求实际走了哪套身份
- UI 上看似“分账号管理”，运行时却不是账号隔离

这正是本需求要解决的问题。

## 3. 必须满足的规则

1. 一个账号只能绑定一套身份包。
2. 一个身份包只能绑定一个账号。
3. 未绑定身份包的账号不得真实出站。
4. 真实请求不得静默回落到全局默认代理、默认 UA 或默认 TLS 行为。
5. 系统必须能够证明某次请求实际使用了账号绑定的身份包。

## 4. 最小产品能力

系统至少要具备四类能力：

### 账号管理

- 展示账号
- 展示账号是否已绑定身份包
- 支持绑定、换绑、解绑

### 身份包管理

- 创建或导入身份包
- 编辑身份包基础信息
- 查看身份包是否已被占用

### 运行时强制执行

- 在真实请求链路中识别当前请求属于哪个账号
- 按账号绑定关系选择对应身份包
- 当账号未绑定、绑定冲突或身份包不可用时，直接拒绝请求

### 证据与验证

- 为请求记录账号、身份包、出口信息
- 能区分“只是配置了绑定”与“真实请求已经按绑定生效”

## 5. 明确不接受的伪实现

以下做法不算完成需求：

- 只在 UI 上展示绑定关系，但运行时仍走全局出口
- 只改代理，不管 UA 和 TLS
- 未绑定时仍允许请求继续出站
- 通过猜测请求内容来判断账号
- 只记录计划配置，不记录真实请求证据

## 6. 分阶段要求

### 第一阶段

先完成账号、身份包、绑定关系的模型和管理界面。

### 第二阶段

把绑定关系真正接入运行时请求链路，做到按账号强制使用绑定身份。

### 第三阶段

补齐请求级证据和验证能力，能够证明绑定真实生效。

## 7. 验收标准

只有同时满足下面几点，才能认为需求完成：

1. 系统中每个账号都能看到唯一绑定身份包。
2. 未绑定账号的真实请求会被阻断。
3. 已绑定账号的真实请求会按绑定身份出站。
4. 系统可以查询某次请求对应的账号、身份包和出口证据。
5. 文档和 UI 不会把“本地配置完成”误说成“运行时已生效”。

## 8. 适用方式

把这份 PRD 用到其他类似项目时，只需要替换各项目自己的账号实体名、运行时索引名和代理组件名即可。

如果一个项目做不到“运行时按账号识别并强制选择身份包”，那它最多只完成了第一阶段，不能宣称已经满足本需求。
